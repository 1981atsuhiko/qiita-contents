# qiita-articles 運用・執筆ガイド

> **Note:** Claude Codeが実際に読み込む実行用の指示は `../.claude/CLAUDE.md`（プロジェクトルート）にも複製している。本ファイルを更新した場合は同内容を反映すること。

このリポジトリは [Qiita CLI](https://github.com/increments/qiita-cli)（`@qiita/qiita-cli`）で記事をGit管理するためのもの。
コマンドの使い方は [README.md](README.md) を参照。ここでは**記事執筆時に守るべきルール**をまとめる。

## ファイル運用ルール

- 記事ファイルは `public/` 配下に **2桁連番 + `.` + 識別用スラッグ** で保存する（例: `04.ses-engineer-future-management.md`）
  - 連番は執筆・公開順。ファイル名と Qiita 記事ID（フロントマターの `id`）は無関係なので、リネームしても公開済み記事との紐付けは保たれる
  - 新規作成は `npx qiita new <slug>` → `public/` 内で既存ファイルの最大連番+1を付けてリネーム → 編集、の順で進める
- `npx qiita ...` コマンドは必ず `qiita-articles` ディレクトリ内で実行する（親ディレクトリで実行すると実行ファイルが見つからずエラーになる）
- `npx qiita new` はローカルにファイルを作るだけで Qiita への通信は発生しない。Qiitaへの反映は `npx qiita publish <basename>` のみ
  - **新規投稿の反映**: フロントマターの `id` が `null` の状態で `npx qiita publish <basename>` を実行 → 新規POSTされ、`id` が自動で書き込まれる
  - **記事更新時**: 既存記事（`id` が入っている状態）を編集後、同じく `npx qiita publish <basename>` を実行 → PATCH更新される。新規投稿時と同じコマンドで良く、`id` の有無で動作が自動的に切り替わる

### 著者の立場・視点

- 著者は **SES企業のエンジニアリングマネージャー**。一人称は「私」、現場感のある実務目線で書く
- 断定しすぎず「〜と考えています」「〜という見立てです」のように、あくまで個人の見解として書く

### 記事構成のパターン

1. `## はじめに` — 前回記事へのリンクと要約 → 今回のテーマ提示
2. `## 1. 〜` `## 2. 〜` のように数字見出しで論点を分割（3〜4個程度）
3. 必要に応じて比較表（Markdownテーブル）で「これまで」と「今後」を対比させる
4. `## おわりに` — 今回の要点を1〜2文で振り返り、**次回の予告**で締める（次回予告は実際に次の記事で回収すること）

### フロントマター

```yaml
tags:
  - ses
  - キャリア
  - コラム
  - エンジニアリングマネージャー
  - 生成AI
private: false
```

- タグは連載共通でこの5つを使う
- `title` は「SES業界という特殊な立ち位置から見た生成AI論 第N回 〜サブタイトル〜」の形式
- 前回記事へのリンクは **ローカルファイルパスではなく、実際のQiita記事URL** を使う
  - URL形式は `https://qiita.com/1981atsuhiko/items/<id>`（`<id>`はリンク対象記事ファイルのフロントマターの`id`の値）
  - リンクを貼る前に、対象ファイル（例: `public/0N.basename.md`）を読んで`id`を確認してから本文に埋め込むこと
  - `id`が`null`（まだ未公開）の記事にはリンクしない。公開済みの記事にのみリンク可能

## 投稿時のリンク切れチェック

`npx qiita publish` で記事を公開・更新した後は、必ず以下を行う。

1. `public/` 配下を連番でソートし、**直近5件**の記事ファイルを対象にする
2. 各ファイル本文中の全リンク（`[テキスト](URL)`形式）を抽出する
3. 抽出した各URLに実際にアクセスし、200応答であることを確認する（404やアクセス不可のものはリンク切れとして報告する）
4. ローカルファイルパス形式のリンク（`0N.basename.md`のような相対パス）が混入していないかも合わせて確認し、混入していれば上記のQiita記事URL形式に修正する

### 文体・トーン

- 見出し以外は地の文中心。箇条書きは「具体例の列挙」「対比」に使う程度に留め、多用しない
- 強調（`**太字**`）は、その回で一番伝えたい主張・キーワードに絞って使う
- 「前回書いたとおり」「前回触れたとおり」のように、前の回の内容を明示的に参照し、連載としての連続性を出す
